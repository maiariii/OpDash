## 0. Applicability and flexibility contract (read first)

This sheet is optimized for **categorical inventory and monitoring dashboards** (hierarchical grouping, condition/status counts, issue tracking). It is safe to reuse for other dashboard types **only if** you treat everything below as tiered, not uniformly mandatory.

**Precedence rule:** This section governs the rest of the page. Wherever a later section says "must," "required," or "every dashboard," read it as **"must — when the feature applies to the current dashboard type."** Never force a module that does not fit the data, and never drop a dataset-appropriate feature just to satisfy a template requirement.

### Tier 1 — Core (apply to every dashboard)

- Single-file HTML with embedded CSS + JS (unless the user requests components only).
- Visual identity: CSS `:root` variable palette, rounded cards, radial-gradient background, strong typographic hierarchy.
- Layout shell: `.app` single-column main canvas, `.topbar` hero, `.content` cards. Do not include a sidebar unless the user explicitly requests one or the dashboard truly needs separate file/page navigation.
- Sidebar links navigate to distinct pages/screens; multiple files/datasets become sidebar page links, never in-content tabs.
- KPI grid with plain typed titles, oversized numbers, no pill/sticker titles, and no blank cells (last row stretches to fill).
- At least one primary visualization plus a searchable, sortable, filterable, paginated data table and a row-detail modal.
- Safe data loading with schema/shape validation, a loading state, no hardcoded sample rows, and robust text decoding.
- Responsive/mobile layout (sidebar collapses to bottom navigation).

### Tier 2 — Conditional (apply only when the data/dashboard type calls for it)

- Hierarchical drilldown levels + breadcrumbs — only when the data has nested categorical/geographic hierarchy.
- Stacked-bar ↔ heat map toggle — only for categorical composition across groups.
- Condition-share donut/histogram (working/broken/spare, complete/pending, available/unavailable, etc.) — only for status/condition datasets.
- Issue/risk summary — only when issue/risk logic exists in the data.
- "Distribution by" / per-card "Split by" / category-picker modal — only when there are non-location categorical segmentation variables.
- Dependent parent→child filters and auto-detected categorical filters — only when such columns exist.
- CSV-specific loading, `REQUIRED_FIELDS`, header validation, and encoding/mojibake repair — only when the source is CSV. For JSON/API/DB sources, validate the equivalent shape instead.
- Philippine place-name normalization (Las Piñas, Parañaque) — only for datasets containing those accented names; generalize or omit otherwise.

### Tier 3 — Optional enhancements (use when they add value)

- Touch/long-press tooltip behavior, advanced/basic mode toggle, unit/aggregation switch, and alternate drilldown modes.

### Swapping Tier 2 modules for other dashboard types

- **Time-series / trend:** line/area charts, date-range controls, period comparison — replace hierarchical drilldown.
- **Financial / executive scorecard:** variance vs. target, sparklines, waterfall — condition-share may not apply.
- **Geospatial / map:** map + clustering as the primary visualization instead of stacked bars.
- **Funnel / cohort / conversion:** funnel or cohort grid instead of the composition donut.
- **Real-time ops:** live status tiles and refresh instead of a static CSV load.

When a Tier 2 or Tier 3 module does not fit the data, **omit it silently** — do not stub an empty or misleading version, and do not degrade a better-suited choice to match this template.

## 1. AI role and task

You are designing a **single-file HTML inventory and monitoring dashboard** from a structured CSV or tabular dataset.

Your output must be a complete, copy-pasteable HTML file with embedded CSS and JavaScript. It must replicate the dashboard architecture used in <page url="https://app.notion.com/p/389f6dd1f0c88074a55ee023c63a103d">CCTV Inventory</page>:

- No sidebar by default; keep the dashboard full-width unless the user explicitly requests page/file navigation.
- If the user provides multiple files or datasets, represent each file as its own sidebar page link and dedicated main-panel page/screen. Do not place multiple-file tabs or dataset-switching buttons inside the content area or data controls card.
- Branded hero/topbar.
- Data controls card.
- KPI card grid with plain typed KPI headings, oversized numbers, and no sticker-style title pills.
- Drilldown analytics section with chart-type toggle cycling through normal stacked bars and heat map only.
- Composition chart with donut/histogram toggle.
- Issue/risk summary chart with chart-type toggle.
- Searchable, sortable, filterable, paginated data table.
- Clickable row detail modal.
- CSV loader with strict schema validation, no hardcoded sample rows, a loading modal while CSV files are fetched and parsed, and robust text decoding for UTF-8 / Windows-1252 CSV files so names with accented or special characters render correctly.
- Responsive mobile layout.

## 2. Design principles to preserve

### Visual identity

Use the same overall style logic as the CCTV dashboard:

- **Academic Blue / navy foundation** for headers, borders, and primary actions.
- **Gold / amber accent** for highlights and warning states.
- **Green, red, purple, slate** for operational categories.
- Large rounded cards.
- Strong bordered dashboard panels.
- Clean grid spacing.
- No heavy shadows unless used subtly for modals or side navigation.
- A soft radial-gradient background.
- Strong typographic hierarchy with bold labels, compact helper text, and clear card titles.
- Typography: load and use **Plus Jakarta Sans** for headings and high-emphasis UI (titles, KPI numbers, buttons, badges) through a `--font-head` variable, and **DM Sans** for body text, labels, and filters through the base `--font` variable. Do not use IBM Plex Sans.
- KPI cards must use real typed text headings instead of sticker, badge, or pill labels at the top of the card.
- KPI numbers should be visually dominant and prominent, matching the bigger KPI-number treatment used in the Resource Mapping dashboard pattern; use about `clamp(36px,2.5vw,43px)` or an equivalent large-but-balanced scale for standard KPI cards instead of small 20px–30px values. Keep them large enough to dominate the card but not so oversized that the numeric value and its unit wrap onto separate lines. Make sure the KPI number selector out-specifies any generic `.card h2` rule (for example use `.kpi-card .kpi-number`, not a bare `.kpi-number`), so the enlarged size actually applies instead of being overridden by the card heading size.
- Do not repeat the KPI title below the number; the title belongs only at the top of the card.
- Keep KPI cards compact after removing the repeated label, with no unnecessary blank space below the number/helper text.

## 3. Dashboard sections to replicate

### A. Sidebar page navigation

Use four page links unless the dataset clearly requires different labels:

- Overview
- Drilldown
- Records
- Issues

Each nav link must represent a distinct dashboard page/screen, not a section inside the current page.

For multi-file dashboards, replace or extend these links with one dedicated page per file/dataset. For example:

- Personnel
- Enrolment
- Inventory
- Monitoring

Each file link must open a dedicated dashboard page/screen in the main panel. Do not place multiple-file tabs in the data controls card or near the top of the main content.

Rules:

- Use `href` values that point to separate page files/routes, such as `overview.html`, `drilldown.html`, `records.html`, and `issues.html`.
- For multi-file dashboards, use file-specific `href` values such as `personnel.html`, `enrolment.html`, or another clear slug per file.
- Use `data-page`, not `data-view`.
- Do not call `preventDefault()` on sidebar links; the sidebar should navigate.
- Use `body[data-current-page="..."]` or the current pathname to mark the active page.
- If generating a single prototype file, the current file may include the full overview layout, but the sidebar code must still be written as page navigation that can be reused across separate HTML pages.

Example:


The topbar must explain:

### C. Data controls

At minimum include:

- First-level geography/category filter.
- Second-level geography/category filter. The second-level filter (such as division) must depend on the first-level filter (such as region): when a region is selected, the division dropdown only lists divisions that belong to that region, and an invalid division selection resets when the region changes. More generally, each dependent child filter should only offer values that are valid for the currently selected parent filter.
- For geographic datasets, basic mode must include the common location filters when those columns exist: **Region**, **Province**, **Division**, and **Municipality / Legislative District**. Do not hide these common location filters inside advanced mode.
- Put the basic filter controls inside a dedicated nested subcard within the Data Controls card. The location filters must occupy the entire first row of that subcard. Place subsequent non-location filters and toggles inside the same subcard on the next row. Do not show a "Location Filters" heading/text inside the subcard unless the user explicitly requests a section label.
- Use the required geographic drilldown hierarchy: **Region → Province → Division → Municipality / Legislative District**. If both municipality and legislative district columns exist, the default basic-mode selector should be **Municipality**.
- When both **Municipality** and **Legislative District** columns exist, add a compact icon-only toggle in the Municipality selector header so the user can switch that basic-mode field between Municipality and Legislative District without opening advanced mode. Use a simple switch icon such as `⇄`; do not use a text button like "Use LD".
- Drilldown mode selector when there are alternate third-level groupings.
- Status/condition filter.
- Unit/aggregation mode selector.
- Reset button placed beside the Advanced Mode toggle in the data-controls header, not inside the input grid. Do not show a loading/status helper-text line below the controls unless the user explicitly requests it. The Reset button and the Advanced Mode toggle must not display the decorative ⇄ swap-arrow icon that chart-type toggles use; suppress their `::before` icon (for example `#resetBtn::before,#advancedToggle::before{display:none}`) even if they reuse the chart-toggle style.
- Auto-detected categorical filters: when the CSV loads, detect every categorical (non-numeric) column and automatically render a filter dropdown for each categorical attribute column, so no categorical field is left without a filter (for example classification in personnel or gender in enrolment). Keep region and division as filters (with division dependent on region), and exclude purely numeric columns and deep geographic hierarchy levels below division (such as province, legislative district, and barangay), which are handled through drilldown rather than as dropdowns.
- Auto-activate drilldown from filters: selecting a value in the first-level (region) or second-level (division) filter must automatically drill the chart into that group, so it immediately shows the next hierarchy level beneath the selection (for example, selecting a region shows that region's divisions; selecting a division shows the next level beneath it). Infer parent hierarchy levels from the data when a child filter is set without its parent.
- A mode toggle button that expands/collapses the advanced controls inside the data controls card. When advanced controls are hidden (basic mode), label it "Toggle Advanced Mode" and style it yellow/gold. When advanced controls are visible, label it "Toggle Basic Mode" and style it with the normal secondary/blue treatment.
- Advanced controls must be enclosed inside a visually distinct nested card within the data controls area. Use a highlighted treatment (for example amber/gold border or soft amber background) to make the advanced area clearly differentiated, but keep the heading as plain text reading **"Advanced Controls"** in the same navy/font color family as the main **Data Controls** heading. Do not use a pill, badge, button, or sticker-style label for the heading, and do not include explanatory text about advanced mode inside the card. Input labels within the advanced card — especially **Distribution by** and **Units** — must use the same muted label color as basic input titles like **Region** and **Division**, not the advanced accent color.
- Within the advanced controls card, place **Distribution by** before **Units**. As much as possible, advanced-control filters should occupy a single row on desktop/wide layouts; use responsive wrapping only for narrower screens. If both **Municipality** and **Legislative District** columns exist, include a **Jurisdiction** field in the advanced controls card where users can choose whether drilldowns and charts use municipal drilldown or legislative-district drilldown. Omit or hide alternate drilldown controls only when the dataset does not contain alternate jurisdiction fields or when the requested control surface should remain limited.
- "Distribution by": a global control that changes the breakdown/segmentation variable used to divide every chart (stacked-bar segments, donut slices, and histogram bars) into any available non-location categorical variable in the dataset. In the default state (before the user changes "Distribution by"), every chart including the composition donut and the issue/summary histogram must segment by the same single default distribution variable, not different variables per chart. Exclude geography/location/hierarchy fields such as region, province, division, district, municipality, barangay, site, or equivalent location drilldown levels. It does not change the chart axes or drilldown grouping, and it applies to all charts on the shown page. Label the default option with the actual default segmentation variable's real name (for example its field name), not a generic "Default breakdown", and make sure that default variable is not also listed a second time in the dropdown. Instead of showing the active distribution variable as a static tag/sticker, give each chart card (stacked-bar, composition, and issue/summary charts) its own compact "Split by" dropdown placed near that card's chart-type toggle. Each per-card dropdown changes the segmentation variable for only that one card's chart (the other cards are unaffected). When the user changes the global "Distribution by" control in the data controls, every per-card dropdown updates to match the new global selection. Split-by dropdown options must come only from real available dataset/CSV columns that are valid segmentation variables; do not add synthetic options such as "Readiness gaps" unless the source data actually has that column and the user explicitly asks for it. Keep each card's "Split by" dropdown and chart-type toggle together on a single row in the card header (use a non-wrapping flex row); never let the chart-type toggle wrap onto a new line below the dropdown.
- Category picker for distribution variables: provide a compact gear-icon button beside the "Distribution by" control that opens a modal listing every value of the selected segmentation variable, each with a checkbox and its total. The user can choose exactly which values appear across all charts (for example show only 5 of hundreds of positions). Include quick actions such as Top 5, Select all, and Clear on the left side of the modal toolbar. Put the sort toggle on the opposite/right side, styled distinctly from the other quick-action buttons, and let it switch the category list between sorting by value (descending) and alphabetical (A–Z) order. Include a debounced search field inside the modal that filters the category list while typing or backspacing; checked selections must persist when the search filter is cleared or changed. For any segmentation variable with more than 5 distinct values, automatically default to showing only the top 5 by value until the user customizes the selection.
- "Units": changes the displayed metric. Provide a record-count option (labeled for the dataset's entity, such as "Number of schools" or "Number of records") and one "Sum of <numeric field>" option per numeric variable in the dataset (such as "Sum of learners"). Default to the primary numeric variable, and keep KPI headline values fixed on their own metric regardless of the selected unit.

Do not use the data controls card for switching between multiple files. File/dataset switching belongs in the sidebar as dedicated page links only.

Adapt the controls to the dataset. If the dataset is not geographic, use a category hierarchy such as:

- Department → Team → Asset Type → Asset Group
- Region → Division → Municipality → Site
- Program → Component → Activity Type → Location
- Category → Subcategory → Owner → Item Group

### D. KPI grid

The KPI grid shows dataset totals plus the active distribution breakdown:

1. One record-count card for the dataset, labeled specifically for the entity being viewed (for example "Personnel Records" or "Enrolment Records"). Do not use a generic label such as "Total rows".
2. One card per numeric column in the dataset showing the sum/total of that column (for example total headcount or total learners).
3. One card per value of the currently active "Distribution by" category (defaulting to the chart's default segmentation variable), each showing that category's nationwide/in-view total of the primary numeric metric (for example total male and total female learners, or one card per grade level). Respect the category picker: show only the selected/visible categories (top 5 when auto-limited).

KPI grid layout:

- Prefer a single row of KPI cards.
- When there are too many cards for one row, split them into the fewest additional rows and distribute the cards as evenly as possible across rows.
- Rows may be uneven in count, but there must be no blank/empty cells in any row; the last row's cards stretch to fill the full row width (use a flex layout with growing cards rather than a fixed grid that leaves gaps).

KPI headline values stay fixed on their own metric regardless of the selected "Units" option.

Each KPI card must include:

- Plain typed KPI title at the top of the card. Do not use sticker-style tags, badges, or pills for the KPI title.
- Prominent, oversized numeric value (about `clamp(36px,2.5vw,43px)`), large enough to dominate the card without wrapping the value and its unit onto separate lines.
- Short helper text.
- No repeated title/label below the number.

### E. Drilldown analytics

Implement a drilldown sequence.

For geographic datasets, the required sequence is:

1. Region
2. Province
3. Division
4. Municipality or Legislative District

If both municipality and legislative district columns exist, default to Municipality and provide a Jurisdiction control/toggle to switch to Legislative District.

For a generic dataset, define this as:

```jsx
const DRILLDOWN_LEVELS = [
  { key:"level1", label:"Level 1" },
  { key:"level2", label:"Level 2" },
  { key:"level3", label:"Level 3" },
  { key:"level4", label:"Level 4" }

```

The chart must support:

- Normal stacked bar view, where total bar length is proportional to the largest group total.
- Heat map view rendered as a simple data table (not separated blocky cells/cards): one row per group, one column per category plus a Total column, with each value cell tinted by a continuous gradient based on its value.
- A chart-type toggle that cycles only this drilldown card through `stacked` → `heatmap`.
- Keep the stacked bar compact, with tight spacing between rows and slim bar height.
- The numeric value labels inside the stacked bar segments use an enlarged, clearly readable font size that is larger than the default segment label size.
- Segment label text must automatically switch for contrast: use white text on dark bar colors, and black/dark navy text on light bar colors such as yellow, pale green, or light orange.
- When a segment is too small/narrow to legibly fit its value, omit that segment's inline label instead of overflowing or clipping it.
- Data-label visibility must be based on the segment's **absolute rendered width relative to the full chart track**, not only its percentage share inside its own bar. This prevents inconsistent labeling where a smaller lower bar shows a value but a wider upper bar hides one.
- On desktop/fine-pointer devices, hovering a bar shows a styled tooltip/popover at the pointer location and above the bars that lists the breakdown count for only the currently selected/filtered categories plus the total (do not append the unselected categories), using card-style CSS that matches the dashboard (rounded border, soft shadow, navy text).
- On tablet/touch devices, do not rely on sticky hover. Short tap drills down; long-press shows a temporary tooltip preview; releasing the long-press hides the tooltip and must not also drill down. Suppress the browser's native long-press context menu / touch callout on bar rows so the dashboard tooltip appears instead of right-click-style browser options.
- Stacked bar row labels and numeric end labels use the standard default label size.
- Clickable rows/cells to drill down.
- Breadcrumb trail.
- Grouped aggregations.

Heat map column rule:

- Break the selected group down by the same category/composition variable shown in the stacked bar segments, so the heat map columns reflect the currently selected category breakdown (for example one column per status, grade level, or condition value) plus a total column. Do not fall back to unrelated generic summary columns such as records, top segment, or share when category columns are available.

Heat map layout and sorting:

- Render the heat map as a clean, simple table rather than a grid of separated blocky cells/cards: minimal borders, tight cells, one row per group and one column per category plus a Total column.
- Color each value cell with a continuous four-stop gradient based on its value (computed within that column's own range): green for low values, yellow for lower-middle values, orange for upper-middle values, and red for the highest values; the group/label column stays uncolored.
- Make every column header a sort control so the user can sort the rows by that column (group name alphabetically, or any category/total column by value), toggling ascending/descending and showing the active sort direction.

Heat map color rules:

- Every visible matrix column must receive a heat color, including label/group, totals, records, category values, issue values, and share/percentage columns.
- Use a continuous gradient, not only 2–3 static classes. The shade must communicate low, middle, and extreme values within that column's own range.
- Compute each column's min/max independently so the user can compare extremes and middle values within that metric.
- For the heat-map table, use the same four-stop value palette across columns: green → yellow → orange → red, where green represents the lowest values and red represents the highest values in that column.
- Keep text legible by switching text to white on dark extreme cells.

### Touch / tablet tooltip rule

For stacked bar rows:

- Short tap = drill down.
- Long press, around `500–550ms`, = show tooltip preview.
- Release after long press = hide tooltip and do **not** drill down.
- Movement beyond a small threshold, around `12px`, cancels the long press.
- Mouse / fine-pointer devices keep ordinary hover behavior.
- Use pointer events (`pointerdown`, `pointermove`, `pointerup`, `pointercancel`, `pointerleave`) rather than separate mouse and touch handlers.
- Suppress native context menus on bar rows with both CSS and `contextmenu` event handling.

Required CSS for bar rows:

```css
.bar-row,
.bar-row *{
  -webkit-touch-callout:none;
  -webkit-user-select:none;
  user-select:none;
  -webkit-tap-highlight-color:transparent;
}
```

Reference implementation pattern:

```jsx
const LONG_PRESS_MS = 520;
const LONG_PRESS_MOVE_LIMIT = 12;
let longPressTimer = null;
let longPressActive = false;
let longPressStartX = 0;
let longPressStartY = 0;

function isHoverDevice(){ return window.matchMedia("(hover: hover) and (pointer: fine)").matches; }
function isTouchLikePointer(e){ return e && (e.pointerType === "touch" || e.pointerType === "pen"); }
function clearLongPressTimer(){ if(longPressTimer){ clearTimeout(longPressTimer); longPressTimer = null; } }

function suppressBarContextMenu(e){
  if(e && e.preventDefault) e.preventDefault();
  if(e && e.stopPropagation) e.stopPropagation();
  return false;
}

function startBarPress(e,el){
  if(!isTouchLikePointer(e)) return;
  if(e.preventDefault) e.preventDefault();
  clearLongPressTimer();
  longPressActive = false;
  longPressStartX = e.clientX;
  longPressStartY = e.clientY;
  longPressTimer = setTimeout(() => {
    longPressActive = true;
    showBarTooltip({clientX:longPressStartX,clientY:longPressStartY},el.dataset.tooltip);
  },LONG_PRESS_MS);
}

function moveBarPress(e,html){
  if(isTouchLikePointer(e)){
    const dx = Math.abs(e.clientX - longPressStartX);
    const dy = Math.abs(e.clientY - longPressStartY);
    if(!longPressActive && (dx > LONG_PRESS_MOVE_LIMIT || dy > LONG_PRESS_MOVE_LIMIT)){
      cancelBarPress();
      return;
    }
    if(longPressActive) showBarTooltip(e,html);
    return;
  }
  if(isHoverDevice()) showBarTooltip(e,html);
}

function endBarPress(e,level,name){
  if(isTouchLikePointer(e)){
    clearLongPressTimer();
    if(longPressActive){
      hideBarTooltip();
      longPressActive = false;
      if(e.preventDefault) e.preventDefault();
      return;
    }
    setDrill(level,name);
    return;
  }
  setDrill(level,name);
}

function cancelBarPress(){
  clearLongPressTimer();
  longPressActive = false;
  hideBarTooltip();
}

document.addEventListener("contextmenu",e => {
  if(e.target && e.target.closest && e.target.closest(".bar-row")){
    suppressBarContextMenu(e);
  }
});
```

Bar row elements should include:

```html
oncontextmenu="return suppressBarContextMenu(event)"
```

### Stacked-bar label consistency rule

For stacked bars, data labels should activate based on the actual visual width available to that segment:

```jsx
const trackWidth = Math.max(groupTotal / maxGroupTotal * 100,4);
const segmentShare = segmentValue / groupTotal * 100;
const segmentWidth = trackWidth * segmentShare / 100;
const label = segmentWidth >= 7 ? fmt(segmentValue) : "";
```

Adjust the `7` threshold per design density, but keep the rule based on absolute rendered width. Do not use only `segmentShare >= X`, because that creates inconsistent labels across bars of different total lengths.

- Unavailable
- At risk
- Resolved

### G. Issue / summary histogram

Provide an issue histogram/donut toggle only when issue/risk dimensions are actual fields in the source data or are explicitly requested. Otherwise, use this side-by-side card as a second composition view that follows the same CSV-backed segmentation variables as the donut by default.

Rules:

- On initial load, the donut and histogram must talk about the same selected distribution variable.
- The histogram's Split-by dropdown must list only available dataset/CSV columns that are valid non-location categorical segmentation variables.
- Do not include a synthetic "Readiness gaps" or missing-field option unless that field exists in the source data and the user explicitly requests it.

### H. Data table

The table must include:

- Search input.
- Sortable headers.
- Per-column filters.
- Pagination.
- Row click to open modal.
- Compact badges for statuses.
- Display all dataset/CSV columns by default (one column per source field), plus any derived status/attention column, unless the user explicitly asks for a subset.
- Freeze/pin the record identity columns (ID and name) so they stay visible during horizontal scroll: place them as the first two columns, give them fixed widths, and make them `position:sticky` with `left` offsets (`left:0` and `left:<id width>`), an opaque background, and a higher stacking order than the scrolling cells. Give the table a `min-width` (or per-column widths) wider than its container so the remaining columns scroll horizontally beneath the frozen identity columns.
- Performance for large datasets (tens of thousands of rows): re-render only what changed. A control that only affects the charts (such as "Distribution by") should re-render just the KPIs and charts, not the full records table. Reuse a single cached `Intl.Collator` for sorting instead of calling `localeCompare` with options on every comparison, and memoize expensive per-column computations (such as distinct category values) that do not change after load.

Required table functions:

- `tableColumnValue(row,key)`
- `applyTableColumnFilters(rows)`
- `sortTableRows(rows)`
- `renderTable(rows)`
- `renderTableControls(totalRows,totalPages,start,count)`

- Record title/name.

### Schema validation rule

Every generated dashboard must validate the input CSV headers before rendering.

Use:

```jsx
function validateCsvHeaders(headers){
  const normalized = headers.map(h => h.trim().toLowerCase());
  const missing = REQUIRED_FIELDS.filter(field => !normalized.includes(field));
```

### CSV encoding and special-character handling

Every generated dashboard must load CSV files through `arrayBuffer()` and decode them explicitly instead of relying only on `response.text()`.

Required behavior:

- Decode the same bytes as UTF-8 and Windows-1252.
- Score both decoded strings for replacement characters and common mojibake patterns.
- Use the cleaner decoded string.
- Strip a leading BOM.
- Repair common mojibake sequences such as `Ã±`, `Ã‘`, `Â`, `â€“`, and curly quote artifacts.
- Normalize known high-risk place/entity names when the dataset contains accented names. For Philippine education datasets, explicitly normalize `Las Piñas` and `Parañaque` so duplicate mojibake variants do not appear as separate chart groups.
- Apply this cleaning before validation, grouping, filtering, KPI computation, chart rendering, and table rendering.

Reference implementation pattern:

## 5. Replication workflow for AI

When asked to create another dashboard using this pattern, follow this workflow:

1. **Inspect source data**
    - Identify all fields.
    - Confirm required columns.
    - Identify hierarchy fields.
    - Identify numeric fields.
    - Identify status fields.
    - Identify text/remarks fields.
2. **Define normalized row model**
    - Convert raw CSV rows into consistent keys.
    - Treat blank/empty cells as null. Do not substitute a placeholder such as `"Unassigned"`, and exclude null values entirely from all summaries, breakdowns, segment composition, chart categories, KPI breakdown cards, and filter option lists. Only label blanks with a value like `"Unassigned"` if the user explicitly requests it.
    - Convert numeric fields safely.
    - Normalize yes/no/N/A/blank fields.
    - Compute `issue`.
    - Compute `noInventory`.
3. **Define KPI logic**
    - Total records.
    - Total inventory/quantity.
    - Positive/available/working metric.
    - Negative/broken/risk metric.
    - Spare/pending/secondary metric.
    - Attention/issue metric.
    - Missing/no-inventory metric.
4. **Define filters**
    - Top-level filter.
    - Second-level filter.
    - Drilldown mode if applicable.
    - Status/risk filter.
    - Unit mode filter.
    - If multiple files are provided, define one sidebar page per file instead of a dataset tab control.
5. **Build complete single-file HTML**
    - Embedded CSS.
    - Embedded JavaScript.
    - CSV parser.
    - Header validation.
    - Normalization.
    - Aggregation.
    - Rendering functions.
    - Modal functions.
    - Event listeners.
    - Responsive CSS.
6. **Never output partial snippets only**
    - The final answer must include a complete HTML file unless the user asks for a component only.

```html
function parseCsv(text){
  const rows = [];
  let row = [], cell = "", quote = false;

  for(let i=0;i<text.length;i++){
    const ch = text[i], next = text[i+1];

    if(ch === '"' && quote && next === '"'){
      cell += '"';
      i++;
      continue;
    }

    if(ch === '"'){
      quote = !quote;
      continue;
    }

    if(ch === "," && !quote){
      row.push(cell);
      cell = "";
      continue;
    }

    if((ch === "\n" || ch === "\r") && !quote){
      if(ch === "\r" && next === "\n") i++;
      row.push(cell);
      if(row.some(v => String(v).trim() !== "")) rows.push(row);
      row = [];
      cell = "";
      continue;
    }

    cell += ch;
  }

  if(cell || row.length){
    row.push(cell);
    if(row.some(v => String(v).trim() !== "")) rows.push(row);
  }

  if(!rows.length) return [];

  const headers = rows.shift().map(h => h.trim());
  validateCsvHeaders(headers);

  return rows.map(values => {
    const obj = {};
    headers.forEach((h,i) => obj[h] = (values[i] ?? "").trim());
    return normalizeRow(obj);
  });
}

function validateCsvHeaders(headers){
  const normalized = headers.map(h => h.trim().toLowerCase());
  const missing = REQUIRED_FIELDS.filter(field => !normalized.includes(field));
  if(missing.length){
    throw new Error("Missing required field(s): " + missing.join(", "));
  }
}

function normalizeRow(r){
  /*
  EDIT THIS FUNCTION FOR YOUR DATASET.

  Map raw CSV fields into the standard row shape:
  - id
  - name
  - level1 to level4
  - working, broken, spare
  - statusA, statusB, statusC
  - remarks
  - updated_at
  - noInventory
  - issue
  */

  const working = number(r.working);
  const broken = number(r.broken);
  const spare = number(r.spare);
  const remarks = r.remarks || "";
  const statusA = status(r.status_a || r.statusA || "");
  const statusB = status(r.status_b || r.statusB || "");
  const statusC = status(r.status_c || r.statusC || "");

  const noInventory = working === 0 && broken === 0 && spare === 0;
  const textAttention = /repair|replace|broken|risk|issue|urgent|missing|failed/i.test(remarks);
  const issue = broken > 0 || statusA === "no" || statusB === "no" || statusC === "no" || textAttention;

  return {
    id: cleanText(r.id, "No ID"),
    name: cleanText(r.name, "Unnamed record"),
    level1: cleanText(r.level1 || r.region || r.category, "Unassigned"),
    level2: cleanText(r.level2 || r.division || r.subcategory, "Unassigned"),
    level3: cleanText(r.level3 || r.municipality || r.group, "Unassigned"),
    alternateLevel3: cleanText(r.alternateLevel3 || r.legislative_district || r.alt_group, "Unassigned"),
    level4: cleanText(r.level4 || r.school_district || r.site, "Unassigned"),
    working,
    broken,
    spare,
    statusA,
    statusB,
    statusC,
    remarks,
    updated_at: r.updated_at || r.last_updated || "",
    noInventory,
    issue
  };
}

function cleanText(v,fallback="Unassigned"){ return String(v || fallback).trim() || fallback; }
function number(v){ return Math.max(0, Number(String(v ?? "0").replace(/[^\d.-]/g,"")) || 0); }
function status(v){
  const s = String(v || "").trim().toLowerCase();
  if(["yes","true","1","ok","pass","passed"].includes(s)) return "yes";
  if(["no","false","0","fail","failed"].includes(s)) return "no";
  if(["n/a","na","not applicable"].includes(s)) return "n/a";
  return "blank";
}
function labelStatus(s){ return s === "yes" ? "Yes" : s === "no" ? "No" : s === "n/a" ? "N/A" : "Blank"; }
function fmt(v){ return Number(v || 0).toLocaleString(); }
function pct(v){ return Math.round(v || 0) + "%"; }
function escapeJs(s){ return String(s || "").replace(/\\/g,"\\\\").replace(/'/g,"\\'"); }

function filteredRows(){
  const q = recordSearch.value.trim().toLowerCase();
  const level1 = level1Select.value;
  const level2 = level2Select.value;
  const statusFilter = statusSelect.value;

  return rawRows.filter(r => {
    if(level1 !== "all" && r.level1 !== level1) return false;
    if(level2 !== "all" && r.level2 !== level2) return false;
    if(drill.level1 && r.level1 !== drill.level1) return false;
    if(drill.level2 && r.level2 !== drill.level2) return false;
    if(drill.level3 && r[level3Mode] !== drill.level3) return false;
    if(drill.level4 && r.level4 !== drill.level4) return false;

    if(statusFilter === "hasWorking" && r.working <= 0) return false;
    if(statusFilter === "hasBroken" && r.broken <= 0) return false;
    if(statusFilter === "needsAttention" && !r.issue) return false;
    if(statusFilter === "noInventory" && !r.noInventory) return false;

    if(q){
      const haystack = [r.id,r.name,r.level1,r.level2,r.level3,r.alternateLevel3,r.level4,r.remarks].join(" ").toLowerCase();
      if(!haystack.includes(q)) return false;
    }

    return true;
  });
}

function aggregate(rows,key){
  const map = new Map();

  rows.forEach(r => {
    const name = r[key] || "Unassigned";

    if(!map.has(name)){
      map.set(name,{
        name,
        records:0,
        working:0,
        broken:0,
        spare:0,
        workingRecords:0,
        brokenRecords:0,
        spareRecords:0,
        inventoryRecords:0,
        issue:0,
        noInventory:0,
        statusANo:0,
        statusBNo:0,
        statusCNo:0
      });
    }

    const g = map.get(name);
    g.records++;
    g.working += r.working;
    g.broken += r.broken;
    g.spare += r.spare;
    if(r.working > 0) g.workingRecords++;
    if(r.broken > 0) g.brokenRecords++;
    if(r.spare > 0) g.spareRecords++;
    if(r.working > 0 || r.broken > 0 || r.spare > 0) g.inventoryRecords++;
    if(r.issue) g.issue++;
    if(r.noInventory) g.noInventory++;
    if(r.statusA === "no") g.statusANo++;
    if(r.statusB === "no") g.statusBNo++;
    if(r.statusC === "no") g.statusCNo++;
  });

  return Array.from(map.values()).sort((a,b) => a.name.localeCompare(b.name));
}

function totals(rows){
  return rows.reduce((t,r) => {
    t.records++;
    t.working += r.working;
    t.broken += r.broken;
    t.spare += r.spare;
    if(r.working > 0) t.workingRecords++;
    if(r.broken > 0) t.brokenRecords++;
    if(r.spare > 0) t.spareRecords++;
    if(r.working > 0 || r.broken > 0 || r.spare > 0) t.inventoryRecords++;
    if(r.issue) t.issue++;
    if(r.noInventory) t.noInventory++;
    if(r.statusA === "no") t.statusANo++;
    if(r.statusB === "no") t.statusBNo++;
    if(r.statusC === "no") t.statusCNo++;
    return t;
  },{
    records:0,
    working:0,
    broken:0,
    spare:0,
    workingRecords:0,
    brokenRecords:0,
    spareRecords:0,
    inventoryRecords:0,
    issue:0,
    noInventory:0,
    statusANo:0,
    statusBNo:0,
    statusCNo:0
  });
}

function hexToRgb(hex){
  const clean = hex.replace("#","");
  return {
    r:parseInt(clean.slice(0,2),16),
    g:parseInt(clean.slice(2,4),16),
    b:parseInt(clean.slice(4,6),16)
  };
}

function rgbToHex({r,g,b}){
  return "#" + [r,g,b].map(v => Math.round(v).toString(16).padStart(2,"0")).join("");
}

function mixColor(a,b,t){
  const ca = hexToRgb(a);
  const cb = hexToRgb(b);
  return rgbToHex({
    r:ca.r + (cb.r - ca.r) * t,
    g:ca.g + (cb.g - ca.g) * t,
    b:ca.b + (cb.b - ca.b) * t
  });
}

function gradientColor(value,range,palette){
  const min = Number.isFinite(range.min) ? range.min : 0;
  const max = Number.isFinite(range.max) ? range.max : 0;
  const t = max === min ? 1 : Math.max(0,Math.min(1,(value - min) / (max - min)));

  if(t <= .5){
    return mixColor(palette[0],palette[1],t * 2);
  }
  return mixColor(palette[1],palette[2],(t - .5) * 2);
}

function heatMapStyle(value,range,palette){
  const bg = gradientColor(value,range,palette);
  const min = Number.isFinite(range.min) ? range.min : 0;
  const max = Number.isFinite(range.max) ? range.max : 0;
  const t = max === min ? 1 : Math.max(0,Math.min(1,(value - min) / (max - min)));
  const text = t >= .72 ? "#FFFFFF" : "var(--navy)";
  const border = t >= .72 ? "rgba(8,49,95,.32)" : "rgba(8,49,95,.14)";
  return `background:${bg};border-color:${border};color:${text}`;
}

function valueRange(items,getValue){
  const values = items.map(getValue).filter(v => Number.isFinite(v));
  return {
    min:Math.min(...values,0),
    max:Math.max(...values,0)
  };
}

function isNumericTableColumn(key){ return ["working","broken","spare"].includes(key); }

function applyTableColumnFilters(rows){
  const filters = tableState.filters || {};

  return rows.filter(r => Object.entries(filters).every(([key,raw]) => {
    const filter = String(raw || "").trim().toLowerCase();
    if(!filter) return true;

    const value = tableColumnValue(r,key);

    if(["statusA","statusB","attention"].includes(key)){
      return String(value).toLowerCase() === filter;
    }

    return String(value).toLowerCase().includes(filter);
  }));
}

function sortTableRows(rows){
  const key = tableState.sortKey || "level1";
  const direction = tableState.sortDirection === "desc" ? -1 : 1;

  return rows.sort((a,b) => {
    const av = tableColumnValue(a,key);
    const bv = tableColumnValue(b,key);

    if(isNumericTableColumn(key)){
      return ((Number(av) || 0) - (Number(bv) || 0)) * direction;
    }

    const primary = TABLE_COLLATOR.compare(String(av),String(bv)) * direction;
    if(primary) return primary;

    return TABLE_COLLATOR.compare(String(a.id),String(b.id));
  });
}

function syncTableHeader(){
  document.querySelectorAll("[data-sort-key]").forEach(btn => {

    }
  });
}

function renderTable(rows){
  const list = sortTableRows(applyTableColumnFilters(rows.slice()));
  const totalPages = Math.max(1,Math.ceil(list.length / tableState.pageSize));
  tableState.page = Math.min(Math.max(1,tableState.page),totalPages);

  const start = (tableState.page - 1) * tableState.pageSize;
  const pageRows = list.slice(start,start + tableState.pageSize);

  recordTableBody.innerHTML = pageRows.map((r,i) => `
    <tr class="record-row" onclick="openRecordModal('${escapeJs(r.id)}')" style="animation-delay:${Math.min(i*12,160)}ms">
      <td class="col-freeze col-freeze-1"><b>${r.id}</b></td>

      <td>${r.level1}</td>
      <td>${r.level2}</td>

      <td>${fmt(r.broken)}</td>
      <td>${fmt(r.spare)}</td>
      <td>${statusBadge(r.statusA)}</td>
      <td>${statusBadge(r.statusB)}</td>
      <td>${r.issue ? `<span class="update-age risk">Review</span>` : `<span class="update-age ok">OK</span>`}</td>
    </tr>
  `).join("");

  syncTableHeader();
  renderTableControls(list.length,totalPages,start,pageRows.length);
}

function renderTableControls(totalRows,totalPages,start,count){
  const label = totalRows ? `Rows ${fmt(start + 1)}–${fmt(start + count)} of ${fmt(totalRows)} · Page ${fmt(tableState.page)} of ${fmt(totalPages)}` : "No records";

  const controls = `
    <span class="table-page-label">${label}</span>
    <div style="display:flex;gap:6px;align-items:center;flex-wrap:wrap">
      <select class="select" data-page-size>
        <option value="10"${tableState.pageSize===10?" selected":""}>10 rows</option>
        <option value="25"${tableState.pageSize===25?" selected":""}>25 rows</option>
        <option value="50"${tableState.pageSize===50?" selected":""}>50 rows</option>
        <option value="100"${tableState.pageSize===100?" selected":""}>100 rows</option>
      </select>
      <button class="button secondary" type="button" data-page="prev"${tableState.page<=1?" disabled":""}>Prev</button>
      <button class="button secondary" type="button" data-page="next"${tableState.page>=totalPages?" disabled":""}>Next</button>
    </div>
  `;

  [tableControlsTop,tableControlsBottom].forEach(el => {
    el.innerHTML = controls;

    el.querySelector("[data-page-size]").onchange = e => {
      tableState.pageSize = Number(e.target.value);
      tableState.page = 1;
      renderAll();
    };

    el.querySelector("[data-page='prev']").onclick = () => {
      tableState.page = Math.max(1,tableState.page - 1);
      renderAll();
    };

    el.querySelector("[data-page='next']").onclick = () => {
      tableState.page = Math.min(totalPages,tableState.page + 1);
      renderAll();
    };
  });
}

async function fetchFirstAvailableCsv(){
  const errors = [];

  for(const url of CSV_FILE_CANDIDATES){
    try{
      const response = await fetch(url, { cache:"no-store" });
      if(!response.ok) throw new Error(`HTTP ${response.status}`);
      const text = decodeCsvText(await response.arrayBuffer());
      return { url, text };
    }catch(error){
      errors.push(`${url}: ${error.message || error}`);
    }
  }

  throw new Error(errors.join(" | "));
}

async function loadCsvFromAttachedFile(){
  try{
    if(dataSourceNote) dataSourceNote.textContent = `Loading CSV from the same folder as this HTML: ${CSV_FILE_NAME}...`;

    const loaded = await fetchFirstAvailableCsv();
    rawRows = parseCsv(loaded.text);
    drill = { level1:null, level2:null, level3:null, level4:null };

    if(dataSourceNote) dataSourceNote.textContent = `Loaded ${fmt(rawRows.length)} rows from ${loaded.url}. Missing hierarchy values are grouped as Unassigned.`;
    renderAll();
  }catch(error){
    console.error("Unable to load CSV.", error);
    rawRows = [];

    if(dataSourceNote){
      dataSourceNote.textContent = `Could not load ${CSV_FILE_NAME}. Put the HTML and CSV in the same folder and open them through a local web server, for example: python -m http.server 8000, then visit http://localhost:8000/.`;
    }

    renderAll();
  }
}
```

## 8. Implementation checklist

Before considering the dashboard complete, verify:

- [ ]  The file is a complete HTML document.
- [ ]  CSS variables are defined.
- [ ]  Typography uses Plus Jakarta Sans for headings/high-emphasis UI (via `--font-head`) and DM Sans for body text, labels, and filters (via `--font`), not IBM Plex Sans.
- [ ]  No sidebar is shown unless explicitly requested for page/file navigation.
- [ ]  Sidebar links navigate to distinct pages/screens instead of hiding/showing page sections.
- [ ]  Multiple files/datasets appear as sidebar page links, not as tabs or buttons inside the main content.
- [ ]  Each file/dataset page has its own dedicated main-panel dashboard.
- [ ]  Collapsed sidebar uses `insighted_logo_vertical.png`.
- [ ]  Expanded sidebar uses `insighted_logo_horizontal.png`.
- [ ]  Expanded sidebar logo is fully visible and not clipped.
- [ ]  Sidebar logo has no white background/container behind it.
- [ ]  Mobile sidebar becomes bottom navigation.
- [ ]  Topbar uses the same visual architecture, with the standardized eyebrow: `DEPARTMENT OF EDUCATION | HUMAN RESOURCE AND ORGANIZATIONAL DEVELOPMENT AND INFRASTRUCTURE`.
- [ ]  Controls are wired to filtering logic.
- [ ]  The second-level filter (division) depends on the first-level filter (region): selecting a region limits the division options to that region and resets a now-invalid division selection.
- [ ]  Geographic dashboards include common basic-mode location filters when available: Region, Province, Division, and Municipality / Legislative District.
- [ ]  Basic filters are grouped in a dedicated nested subcard inside Data Controls: location filters occupy the full first row, subsequent non-location filters/toggles appear inside the same subcard on the next row, and no "Location Filters" text is shown unless requested.
- [ ]  Geographic drilldown follows Region → Province → Division → Municipality / Legislative District.
- [ ]  If both municipality and legislative district columns exist, Municipality is the default basic-mode selector and its label/header includes a compact icon-only toggle to switch to Legislative District; do not use text labels like "Use LD" for the toggle.
- [ ]  If both municipality and legislative district columns exist, the advanced controls card includes a Jurisdiction field for selecting Municipal or Legislative District drilldown.
- [ ]  Categorical (non-numeric) columns are auto-detected on CSV load and each categorical attribute gets its own filter dropdown in the data controls (for example classification and gender), excluding numeric columns and deep geographic levels below division.
- [ ]  Selecting a region or division filter auto-activates the drilldown to that group's next hierarchy level.
- [ ]  Data controls include a mode toggle that says "Toggle Advanced Mode" in basic mode and "Toggle Basic Mode" in advanced mode, with the Reset button placed beside it in the data-controls header.
- [ ]  The Reset button and Advanced Mode toggle do not show the decorative ⇄ swap-arrow icon behind their text.
- [ ]  The data controls card does not show a loading/status helper-text line below the filters unless the user explicitly requests it.
- [ ]  The "Toggle Advanced Mode" button state is yellow/gold, while the "Toggle Basic Mode" state uses the normal secondary/blue treatment.
- [ ]  Advanced controls are enclosed in a highlighted nested card inside the data controls area, visually distinct from the basic filters.
- [ ]  The advanced card heading is plain text reading "Advanced Controls" in the same navy/font color family as the main Data Controls heading — not a pill, badge, button, or sticker.
- [ ]  Advanced card input labels, including Distribution by and Units, use the same muted label color as regular input titles such as Region and Division.
- [ ]  The advanced card contains Distribution by before Units, keeps advanced-control filters on one row where space allows, and omits/hides alternate drilldown controls when they are not needed for the dataset.
- [ ]  "Distribution by" globally changes the breakdown/segmentation variable that divides every chart (bar segments, donut slices, histogram bars) without changing the chart axes, excludes location/hierarchy variables, and each chart card has its own "Split by" dropdown near its chart-type toggle that changes only that card's segmentation, with all per-card dropdowns syncing to the global selection whenever it changes. Split-by dropdowns list only real available dataset/CSV columns that are valid segmentation variables — no synthetic "Readiness gaps" option unless the source data has that column and the user explicitly asks for it. The default option is labeled with the actual default variable name (not "Default breakdown") and no variable appears twice in the dropdown.
- [ ]  "Units" switches the displayed metric between record count and sums of numeric variables (for example learners or headcount).
- [ ]  KPI cards show the dataset record count (with a specific entity label, not a generic "Total rows"), the total of each numeric column, and one card per active "Distribution by" category value (for example per gender or per grade level) using the primary numeric metric.
- [ ]  KPI card numbers use a prominent Resource Mapping-style scale, about `clamp(36px,2.5vw,43px)` or equivalent — large enough to dominate the card but not so big the value and unit wrap onto separate lines — and the KPI number selector out-specifies `.card h2` (for example `.kpi-card .kpi-number`).
- [ ]  KPI cards respond to the selected "Distribution by" category and respect the category-picker selection.
- [ ]  KPI cards are distributed as evenly as possible across rows with no blank cells in any row (the last row stretches to fill).
- [ ]  KPI cards update after filters.
- [ ]  Drilldown breadcrumbs work.
- [ ]  Normal stacked bars render with total bar length proportional to the largest group.
- [ ]  Stacked bars are compact (tight row spacing, slim bars) with enlarged readable value labels inside the segments, labels switch to black/dark text on light bar colors for contrast, and labels are omitted when a segment is too small to fit them.
- [ ]  Stacked-bar data labels activate consistently using absolute rendered segment width, not only segment percentage within the row.
- [ ]  Hovering a bar shows a styled tooltip at the pointer location above the bars listing only the selected/filtered categories and the total.
- [ ]  On tablet/touch, bar short-tap drills down, long-press shows a temporary tooltip preview, and releasing the press hides the tooltip without drilling down.
- [ ]  Browser native right-click/context-menu options are suppressed on long-press for stacked bar rows, so the custom dashboard tooltip appears instead.
- [ ]  Distribution variables have a compact gear-icon category-picker modal (checkboxes, totals, debounced search, Top 5 / Select all / Clear, and a value/alphabetical sort toggle) and auto-limit to the top 5 values when there are more than 5.
- [ ]  The category-picker sort toggle is visually distinct from the Top 5 / Select all / Clear buttons and is positioned on the opposite/right side of the modal toolbar.
- [ ]  Side-by-side composition and issue chart cards are equal width, and the donut legend table is compact/space-efficient.
- [ ]  Heat map columns reflect the same category/composition breakdown shown in the stacked bar segments.
- [ ]  Heat map renders as a simple gradient-shaded table (not blocky separated cells) with per-column sorting controls in the header and a green → yellow → orange → red value gradient.
- [ ]  Heat map shades clearly distinguish low, middle, and extreme values within each column range.
- [ ]  Each chart card keeps its "Split by" dropdown and chart-type toggle on a single header row; the chart-type toggle never wraps onto a new line.
- [ ]  Chart toggles work.
- [ ]  Each chart toggle updates only its own card and does not activate or redraw unrelated dashboard charts.
- [ ]  Table search works.
- [ ]  Column filters work.
- [ ]  Table sorting works.
- [ ]  Pagination works.
- [ ]  The records table displays all dataset/CSV columns by default (unless the user requested a subset), plus any derived status column.
- [ ]  The record identity columns (ID and name) are frozen/pinned as the first two sticky columns and stay visible while the other columns scroll horizontally.
- [ ]  Rendering is scoped for performance: chart-only controls (such as "Distribution by") re-render only the KPIs/charts (not the full table), table sorting reuses a single cached Intl.Collator, and distinct category values are memoized after load.
- [ ]  Row modal opens and closes.
- [ ]  Dashboard reads live CSV files from the same folder or configured relative paths and does not keep hardcoded sample rows in the script.
- [ ]  CSV loading uses `arrayBuffer()` plus explicit UTF-8 / Windows-1252 decoding, mojibake repair, BOM stripping, and canonical text cleanup before parsing.

- [ ]  A loading modal appears while CSV files are fetched and parsed before first render.
- [ ]  CSV validation rejects missing required fields.
- [ ]  Blank/empty cells are treated as null and excluded from summaries, breakdowns, chart categories, and filter lists (not labeled `"Unassigned"` unless the user explicitly asks for it).
- [ ]  Numeric parsing is safe.
