# InsightED CSS Design Rules

These rules are the canonical styling standard for InsightED dashboards. They combine the base InsightED visual system from InsightED Screenshot UI Setup Prompt with the dashboard rules and standards from Project Questions.

## 1. Rule precedence

1. **Preserve dashboard content and business logic first.**
    - Do not overwrite, rename, remove, hard-code, or replace current data, labels, filters, cards, chart categories, routes, tabs, modules, or event handlers.
    - Apply the design system as a styling and interaction pass only unless the dashboard rule explicitly requires behavior.
2. **Use the InsightED CSS Design Rules as the starting point.**
    - Typography, palette, card treatment, background, header style, nav treatment, and responsive layout should follow the InsightED visual system.
3. **Dashboard rules and standards override any conflicting base styling.**
    - If a base design rule conflicts with the dashboard rules, follow the dashboard rule.
    - Examples:
        - Sidebar auto-hide overrides the fixed 260px desktop sidebar rule.
        - Stacked bar labels should show numbers only, not category names.
        - Tables must prioritize containment and pagination over decorative spacing.

## 2. Core visual identity

1. **Use the Academic Blue palette.**
    - Navy: `#08315F`
    - Blue: `#075985`
    - Bright blue: `#0284C7`
    - Sky blue: `#7DD3FC`
    - Pale blue: `#E0F2FE`
    - Background blue: `#F0F9FF`
    - Gold: `#FBBF24`
    - Amber: `#D97706`
    - Red: `#B91C1C`
    - Green: `#16A34A`
    - Purple: `#7C3AED`
    - Text: `#0F172A`
    - Muted text: `#64748B`
    - Line/border accent: `#BAE6FD`
2. **Use fixed InsightED typography.**
    - Headings and high-emphasis UI: `Plus Jakarta Sans`
    - Body, labels, filters, nav labels, supporting text: `DM Sans`
3. **Use shared CSS tokens.**

2. **Use the Bold Blobs background treatment.**

## 4. Dashboard shell and sidebar

1. **Use a two-zone dashboard shell.**
    - Desktop: sidebar + main content.
    - Mobile/tablet: main content + fixed bottom navigation.
2. **Use sidebar auto-hide for active dashboards.**
    - The base design may describe a fixed `260px` sidebar.
    - For implementation dashboards, use a collapsed `72px` desktop sidebar that expands to `260px` on hover.
    - This prevents layout conflict between the base sidebar width and the dashboard standard requiring auto-hide.
3. **Desktop app grid should match the sidebar behavior.**
    - If sidebar auto-hide is enabled: `.app { grid-template-columns: 72px 1fr; }`
    - If sidebar auto-hide is intentionally disabled for another product page: `.app { grid-template-columns: 260px 1fr; }`
4. **The sidebar must auto-appear on hover only.**
    - Use `:hover` for expansion.
    - Do **not** use `:focus-within` to keep the sidebar open.
    - Clicking a sidebar tab must not disable or lock the auto-hide behavior.
5. **Blur clicked nav links after switching tabs.**

```jsx
document.querySelectorAll(".nav a").forEach(a =>
  a.addEventListener("click", e => {
    e.preventDefault();
    switchView(a.dataset.view);
    a.blur();
  })
);
```

## 7. Header / topbar

1. **Use the exact InsightED command-card header background.**
    - Header is a rounded rectangular command card with a thin navy/blue outline.
    - The **blue field must be wider than the white field**.
    - The blue field occupies roughly the left **70%** of the header.
    - The white field occupies roughly the right **30%** of the header.
    - A sharp diagonal boundary separates the blue and white fields.
    - A darker navy diagonal wedge appears near the blue/white boundary.
    - Pale gold circular glow shapes sit behind the right-side actions or metric area.
    - Header text sits on the blue field and must be white or pale blue.
    - Right-side pills/buttons sit on the white field.
2. **Keep the header compact and content-safe.**
    - Desktop minimum height: around `118px` to `130px`.
    - Desktop padding: around `28px 44px`.
    - Mobile minimum height: around `98px` to `104px`.
    - Mobile padding: around `15px 23px`.
    - Do not allow decorative shapes to cover, dim, or obscure the title.
    - Use `isolation: isolate` and negative z-index pseudo-elements only when needed.
3. **Preserve actual dashboard title text.**
    - Do not replace app-specific titles with sample titles from the visual prompt.
    - Apply the visual treatment to the current title.
4. **Use this canonical header CSS unless a page requires a documented exception.**

## 8. Filter bar

1. **Place the filter bar directly below the header.**
2. **Use a card-like container matching the card outline system.**
3. **Preserve actual dashboard filters.**
    - Do not force sample filter names if the dashboard uses different filters.
4. **Use compact labels and 42px desktop controls.**
5. **On mobile, filters should wrap into 1–2 columns and the button should span full width when needed.**

## 9. Card outline system

1. **All cards must share the same visual language.**
    - KPI cards
    - Chart cards
    - Table cards
    - Donut cards
    - Histogram cards
    - Filter cards

```css
.card {
  background: var(--card);
  border: 2.5px solid color-mix(in srgb, var(--blue) 64%, var(--navy) 36%);
  border-radius: var(--radius);
  box-shadow: none;
}
```

1. **Avoid conflicts with external frameworks.**
    - Do not use gray Bootstrap-style borders.
    - Do not use heavy shadows.
    - Do not override table cells globally in a way that breaks tables.
    - Avoid global rules such as `th { display:block; }`.

## 10. KPI cards

1. **KPI cards must be one row when possible.**
2. **KPI cards may wrap to two rows if too cramped.**
3. **If KPI cards use two rows, distribute cards evenly.**
4. **Every KPI card must have a hover breakdown.**
    - Nationwide view: show region breakdown.
    - Region view: show division breakdown.
5. **KPI hover cards must not be clipped.**
    - KPI cards may use `overflow: visible`.
    - Hover card should have a high enough `z-index`.

```css
.kpis {

  z-index: 30;
  max-height: 260px;
  overflow: auto;
}
```

## 11. Stacked bar graphs

1. **All stacked bar graph segments must have labels inside.**
2. **Segment labels should show the number/figure only.**
    - Do not repeat category names inside the bar because categories are already reflected in the legend.
3. **Use the legend for category names.**
4. **Maintain readable contrast for the numbers.**
5. **Small segments should still attempt to show the figure without breaking the bar layout.**

```jsx
function stackedSegments(values, total, classes, labels) {
  return values.map((v, i) => {
    const w = total ? Math.max(v / total * 100, 0) : 0;
    return "<div class='segment " + classes[i] + "' style='width:" + w + "%' title='" + labels[i] + ": " + fmt(v) + "'>" + fmt(v) + "</div>";
  }).join("");
}
```

## 12. Bar graph drilldown

1. **All bar graphs must support region-to-division drilldown.**
    - Clicking a region bar filters the dashboard to that region.
    - The graph should then show division breakdown.
2. **Drilldown behavior applies to every dashboard bar graph.**
    - HROD
    - SIIF
    - eSF7
    - NOSCA
    - Infra
    - Summary / cross-category
3. **Do not limit drilldown to only one tab.**
4. **Only region rows should be clickable for region-to-division drilldown.**
    - Division rows do not need another drilldown unless explicitly defined.
5. **Use a reusable helper to avoid inconsistent behavior.**

```jsx
function enableRegionDrilldown(rowEl, rowName) {
  if (selectedRegion() === "all") {
    rowEl.classList.add("drillable");
    rowEl.title = "Click to drill down to division breakdown";
    rowEl.onclick = () => {
      document.getElementById("regionFilter").value = rowName;
      document.getElementById("divisionFilter").value = "all";
      renderAll();
    };
  }
}
```

## 13. Drilldown breadcrumbs

1. **Every drilldown-enabled bar graph card must show breadcrumbs.**
2. **Breadcrumbs should appear inside the card itself or directly above the graph.**
3. **Breadcrumbs must allow users to go back up the hierarchy.**
    - All regions
    - Selected region
    - Selected division, if applicable
4. **Breadcrumbs should be rendered for each graph card, not only the first chart.**

```jsx
function renderDrillTrail(id) {
  const trail = document.getElementById(id);
  if (!trail) return;

  trail.replaceChildren();

  const root = document.createElement("button");
  root.className = "crumb" + (selectedRegion() === "all" ? " active" : "");
  root.type = "button";
  root.textContent = "All regions";
  root.onclick = () => {
    document.getElementById("regionFilter").value = "all";
    document.getElementById("divisionFilter").value = "all";
    renderAll();
  };
  trail.appendChild(root);

  if (selectedRegion() !== "all") {
    const regionCrumb = document.createElement("button");
    regionCrumb.className = "crumb" + (selectedDivision() === "all" ? " active" : "");
    regionCrumb.type = "button";
    regionCrumb.textContent = selectedRegion();
    regionCrumb.onclick = () => {
      document.getElementById("divisionFilter").value = "all";
      renderAll();
    };
    trail.appendChild(regionCrumb);
  }

  if (selectedDivision() !== "all") {
    const divisionCrumb = document.createElement("button");
    divisionCrumb.className = "crumb active";
    divisionCrumb.type = "button";
    divisionCrumb.textContent = selectedDivision();
    trail.appendChild(divisionCrumb);
  }
}
```

## 18. Data tables

1. **Data tables must not overflow the card.**
2. **Tables must stay within card boundaries.**
3. **Use fixed layout, wrapping, and safe word breaks.**
4. **Do not apply global display rules that break table cells.**
5. **Avoid `display:block` on `th` or `td`.**
6. **Center numeric columns; keep text columns left-aligned.**
    - Detect numeric columns per column, not by fixed position, since column sets differ across tabs.
    - A column is numeric only if every non-blank visible value in it parses as a number.
    - Apply centering to the header cell, its data cells, the header's sort button, and the column filter input.

```css
table {
  width: 100%;
  max-width: 100%;
  border-collapse: collapse;
  font-size: 12px;
  table-layout: fixed;
}

th,
td {
  display: table-cell;
  padding: 10px 8px;
  border-bottom: 1px solid #E2E8F0;
  text-align: left;
  vertical-align: middle;
  overflow-wrap: anywhere;
  word-break: break-word;
}

th {
  color: var(--muted);
  font-family: var(--font-heading);
  font-size: 11px;
  font-weight: 950;
  text-transform: uppercase;
  letter-spacing: .13em;
  white-space: normal;
}

th.num-col,
td.num-col {
  text-align: center;
}

  text-align: center;
  justify-content: center;
```

```jsx
function numericColumnFlags(columns, rows, isBlank, parseNumber) {
  return columns.map((_, columnIndex) => {
    let hasValue = false;
    const allNumeric = rows.every(row => {
      const value = row[columnIndex];
      if (isBlank(value)) return true;
      hasValue = true;
      return parseNumber(value) !== null;
    });
    return hasValue && allNumeric;
  });
}
```

## 19. Table pagination

1. **Data tables must show at least 10 entries.**
2. **Tables must include row count options.**
    - 10
    - 25
    - 50
    - 100
3. **Tables must include pagination controls.**
    - Prev
    - Next
4. **If fewer than 10 rows are available, preserve the table layout without causing visual overflow.**

## 20. Line charts

1. **Use compact chart density.**
2. **Use 7-day, 15-day, and 30-day range selectors when requested.**
3. **For multi-line charts, each module/category should have a distinct line and color.**
4. **On hover, the active line should stand out and non-hovered lines should become slightly transparent.**
5. **If an average-only toggle is requested, it should show the average line without removing the ability to return to individual lines.**

## 21. Responsive layout

1. **At widths below around `1000px`:**
    - Main layout becomes single-column.
    - KPI cards become two columns.
    - Chart cards become full width.
    - Filter bar becomes 1–2 columns.
    - Header becomes shorter.
    - Card radius becomes around `14px`.
    - Border width becomes around `2px`.
    - Body gets bottom padding for fixed footer nav.
2. **At very narrow widths:**
    - KPI cards may become one column.
    - Module grids and filter grids may become one column.
    - Tables must remain within cards.

## 22. Conflict prevention checklist

Before finalizing any InsightED dashboard CSS, verify:

1. **Sidebar conflict check**
    - If auto-hide is required, `.app` uses `72px 1fr`, not `260px 1fr`.
    - Sidebar expands on `:hover`, not `:focus-within`.
    - Nav click handler blurs the clicked link.
2. **Table conflict check**
    - Tables use `table-layout: fixed`.
    - `th` and `td` use `display: table-cell`.
    - No global selector forces `th` to `display:block`.
    - Text wraps inside the card.
    - Numeric columns are centered (header, cells, sort button, filter input); text columns stay left-aligned.
3. **Stacked bar conflict check**
    - Segment labels are inside every segment.
    - Segment label text is the number/figure only.
    - Category labels appear in the legend.
4. **Drilldown conflict check**
    - Every bar graph has region click drilldown.
    - Every drilldown graph card has breadcrumbs.
    - Breadcrumbs allow return to all regions or selected region.
5. **KPI conflict check**
    - KPI cards fit in one row or evenly across two rows.
    - Every KPI card has a hover breakdown.
    - Hover breakdown is not clipped by card overflow.
6. **Chart layout conflict check**
    - Histogram and donut cards are evenly distributed in the same row on desktop.
    - They stack cleanly on smaller screens.
7. **Mobile conflict check**
    - Fixed footer nav does not cover content.
    - Body has bottom padding.
    - Nav columns match actual tab count.
8. **Content preservation check**
    - No sample labels, sample metrics, or sample routes from the design prompt replaced real dashboard content.
    - CSS changes did not remove existing event handlers or data bindings.
